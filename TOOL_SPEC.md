# USD/JPY Signal Desk 仕様書

最終更新: 2026-05-31

この文書は、人間・別AIエージェント・別チャットが本ツールをレビュー、改修、引き継ぎしやすくするための文章ベース仕様書です。

## 目的

USD/JPYだけを対象に、デイトレからスイング寄りの売買チャンスを検知する補助ツールです。

このツールは「買えば勝てる/売れば勝てる」と保証するものではありません。無料データを使うため、遅延、欠損、取得失敗、スプレッド拡大、滑り、実際の約定差があり得ます。実取引では必ず少額またはデモで検証してください。

## 対象範囲

- 対象通貨ペア: USD/JPYのみ
- 主な時間軸: 1時間足、4時間足、日足
- 想定スタイル: デイトレから数日程度のスイング
- 運用方法:
  - PCローカル版: Streamlitで `http://localhost:8501`
  - PC不要版: GitHub Actions + GitHub Pages
  - スマホ通知: ntfy

## データソース

| データ | 取得先/方法 | 用途 |
|---|---|---|
| USD/JPY価格 | `yfinance` の `JPY=X` | 1時間足、日足、バックテスト |
| 米国債利回り | 米財務省XML | 米金利フィルター |
| FOMC予定 | FRBサイト | イベントリスク |
| 日銀予定 | 日銀サイト | イベントリスク |
| 米指標予定 | BLS ICS | イベントリスク |
| 手動イベント | `events_usdjpy.csv` | 取得できない予定や重要イベントの補完 |

外部サイトは構造変更やアクセス制限で取得失敗する可能性があります。失敗時は完全に黙殺せず、注意欄に「取得できませんでした」と表示します。

## 分析ロジック概要

### 1. テクニカル分析

主に以下を使います。

- EMA20
- SMA50 / SMA200
- ボリンジャーバンド
- RSI14
- MACD
- ATR14
- ADX14 / DI
- 直近サポート/レジスタンス

### 2. 相場環境判定

4時間足を中心に、以下の相場環境を判定します。

- 上昇トレンド
- 下降トレンド
- レンジ/収縮
- ボラ拡大
- 中立/方向確認中

この判定は `analyze_regime()` と `regime_group_name()` によって行います。

### 3. 米金利フィルター

米10年債利回りの5営業日変化、20営業日変化を見て、USD/JPYの買い材料/売り材料/中立を判定します。

### 4. イベントリスク

FOMC、日銀、米CPI、米PPI、米雇用統計などを見て、イベント前後は信頼度に上限をかけます。

手動イベントは `events_usdjpy.csv` に追加できます。

```csv
date,title,impact
2026-06-05,米雇用統計,high
```

`impact` は `high` / `medium` / `low` のいずれかです。空欄や不正値は `high` として扱います。

## 売買判定

`build_setup()` が以下を作ります。

- `bias`: `buy` / `sell` / `wait`
- `direction`: 画面表示用の判定
- `score`: 信頼度目安
- `entry_low` / `entry_high`: 候補ゾーン
- `stop`: 損切り候補
- `target_1` / `target_2`: 利確候補
- `reasons`: 判断根拠
- `warnings`: 注意点

基本スコアはテクニカル、相場環境、米金利、イベントリスクを合成して作ります。

## 実戦用フィルター

`fx_filters.py` の `LiveFilterEngine` が、通知直前に本命/候補/見送りを調整します。

### 入力

`LiveFilterSnapshot` に以下を渡します。

- 売買方向
- 信頼度
- 相場環境
- トレンド/レンジ等のグループ
- 時間帯
- 米金利方向
- イベントリスク
- リスクリワード
- 注意点の数

### 出力

`LiveFilterDecision` が以下を返します。

- `main`: 本命
- `candidate`: 候補
- `none`: 見送り

### 判断思想

過学習を避けるため、過去データの細かいパターンを直接丸暗記しません。

代わりに、以下のような広い条件で実戦向けに調整します。

- 重要イベント前後は見送り、または厳格化
- 薄商い時間帯は厳格化
- 相場環境が中立なら厳格化
- レンジではだましを警戒
- 売買方向と相場環境が逆なら候補に降格しやすくする
- 売買方向と米金利方向が逆なら厳格化
- リスクリワードが不足する場合は見送り
- 注意点が多い場合は本命条件を厳格化

## 通知仕様

### 通知タイプ

| 種類 | 条件 | 回数制限 |
|---|---|---|
| 本命 | 実戦フィルターで `main` | 同じセットアップはクールダウンあり |
| 候補 | 実戦フィルターで `candidate` | 回数制限なし |
| 日次チェック | その日まだ通知がない場合 | 1日1回 |

候補通知は、ユーザー要望により回数制限を設けません。

### クールダウンキー

本命通知は `signal_key()` で同一セットアップ判定を行います。

単純な `buy` / `sell` だけではなく、以下を含めます。

- 売買方向
- スコアの5点刻み
- 候補ゾーン
- 損切り
- 第1利確

これにより、同じ買い方向でも価格帯や水準が変われば別シグナルとして扱います。

## バックテスト仕様

バックテストはPC版の画面で実行します。スマホ版には重い検証UIは出しません。

### 使う前提

- 過去の各時点で、その時点までのデータだけを使って判定を作ります
- 未来の価格は判定に使いません
- 候補ゾーン到達後、利確/損切り/時間切れを検証します

### コスト

以下を成績から差し引きます。

- スプレッド
- 滑り

USD/JPYでは 1 pip = 0.01円 として計算します。

### 約定モデル

`candidate_zone` モードでは、以前は候補ゾーン中心を固定エントリー価格としていました。

現在はより実戦寄りに、バーが候補ゾーンへ触れた時の価格として扱います。

- 買い: 上からゾーンに落ちてきた場合はゾーン上限寄り
- 売り: 下からゾーンに上がってきた場合はゾーン下限寄り
- 始値がゾーン内なら始値

これは完全なティック再現ではありませんが、中心価格固定より保守的です。

### 同一バーで利確/損切りに届いた場合

1時間足だけでは、バー内で利確と損切りのどちらが先に到達したか完全には分かりません。

そのため以下を選べます。

- 保守的に損切り優先
- ローソク足の流れで推定

標準は「保守的に損切り優先」です。

### 検証表示

バックテストでは以下を見られます。

- 取引数
- 勝率
- 合計R
- 平均R
- 最大DD
- PF
- 最大連敗
- 買い/売り別成績
- トレンド/レンジ別成績
- 時間帯別成績
- 過去最適化側/未使用期間側の分割テスト
- 近い設定の比較

## 設定

ローカル版は `notification_settings.json` を使います。

主な項目:

- `enabled`: 通知の有効/無効
- `topic`: ntfyトピック
- `score_threshold`: 本命の基本信頼度
- `candidate_score_threshold`: 候補の基本信頼度
- `daily_summary_hour`: 日次チェック開始時刻
- `live_filter_enabled`: 実戦フィルターの有効/無効
- `interval_minutes`: ローカル監視間隔
- `cooldown_minutes`: 本命通知のクールダウン
- `notify_during_high_event`: 重要イベント前後も通知するか

読み込み時に型チェックと範囲補正を行います。

## 既知の限界

- 無料データのため、価格データが遅延または欠損する可能性があります
- yfinanceのUSD/JPYデータは業者の実際のBid/Askとは異なります
- BLSなどの外部予定表はアクセス制限やサイト変更で取得できない場合があります
- バックテストはティック単位の約定を再現しません
- 重要イベントの時刻は日付ベースの扱いが中心です
- このツールは利益を保証しません

## レビュー時に見るべきポイント

レビューする人やAIエージェントは、特に以下を確認してください。

1. `build_setup()` のスコア配分が過剰に楽観的でないか
2. `LiveFilterEngine` が候補を落としすぎていないか、または甘すぎないか
3. `run_backtest()` の約定前提が実運用の注文方法と一致しているか
4. 外部データ取得失敗時にユーザーへ十分な注意が出るか
5. 通知頻度がユーザーの意図に合っているか
6. GitHub Actionsの無料枠を無駄に消費していないか
7. 新しい分析条件を入れる時、過去最適化だけで採用していないか

## 改修方針

今後の改修では、以下を優先します。

- UIはシンプルに保つ
- 重い検証はバックテスト画面だけで実行する
- 通知直前の判断は軽量な実戦フィルターで行う
- 過去データにだけ合う細かいルールを増やしすぎない
- ロジックを追加する場合は `fx_filters.py` のように責務を分ける
- 実戦反映前に、未使用期間テストで崩れないか確認する
